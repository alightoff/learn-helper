# Roadmap для работы с Codex в VS Code

## 1. Назначение документа

Этот файл объясняет, как правильно работать с Codex внутри VS Code, чтобы разработка шла поэтапно, контролируемо и без хаоса.

Это не ТЗ на продукт. Это инструкция по совместной работе:

- как формулировать задачи;
- в каком порядке идти по разработке;
- какие промпты дают лучший результат;
- как снижать риск переделок.

## 2. Правильная модель взаимодействия

Codex лучше использовать не как "чат для идей", а как разработчика, который уже находится в репозитории и может менять код.

Поэтому хорошие промпты обычно:

- ставят одну конкретную задачу;
- указывают область проекта или нужные файлы;
- описывают ожидаемый результат;
- просят реализовать задачу, а не просто порассуждать, если нужен именно код.

Плохой запрос:

- "Сделай полностью все приложение."

Хороший запрос:

- "Прочитай `01_product_spec.md`, создай стартовый каркас FastAPI-приложения для MVP, добавь базовую структуру папок, конфиг, точку входа и README с запуском."

## 3. Базовые правила постановки задач

### 3.1 Один запрос = один измеримый результат

Один промпт должен приводить к одному понятному результату.

Примеры хороших отдельных задач:

- создать каркас проекта;
- реализовать модели базы данных;
- добавить CRUD для курсов;
- сделать загрузку PDF;
- реализовать заметки;
- добавить таймер.

### 3.2 Всегда привязывай задачу к документу или файлам

Если возможно, указывай:

- какой документ является источником истины;
- какие файлы нужно использовать;
- какие ограничения важно сохранить.

Пример:

- "Используй `01_product_spec.md` как источник истины. Реализуй модели `Course`, `Module`, `Resource` и первую миграцию SQLite."

### 3.3 Всегда проси проверку результата

Хороший инженерный промпт почти всегда должен содержать просьбу:

- проверить запуск;
- прогнать тесты, если они есть;
- кратко описать, что изменено;
- отдельно перечислить ограничения и компромиссы.

### 3.4 Строй проект вертикальными срезами

Не надо сначала просить сделать все модели, потом все роуты, потом все шаблоны, потом весь UI.

Лучше работать вертикальными кусками:

1. сделать тонкий рабочий сценарий;
2. проверить его целиком;
3. перейти к следующему.

Пример вертикального среза:

- создать курс;
- вывести список курсов;
- открыть страницу курса.

## 4. Рекомендуемые фазы разработки

## Фаза 0. Подготовка репозитория

Цель:

- создать чистый каркас проекта и договориться о структуре.

Пример промпта:

```text
Прочитай 01_product_spec.md и создай стартовый каркас локального Python web app для MVP.
Стек:
- FastAPI
- SQLite
- SQLAlchemy
- Alembic
- Jinja2
- HTMX
Нужны:
- структура app/, templates/, static/, data/
- конфиг
- точка входа
- зависимости
- README с запуском
Ничего лишнего не добавляй. После изменений проверь, что приложение стартует.
```

Ожидаемый результат:

- структура проекта;
- файл зависимостей;
- main entrypoint;
- конфиг;
- понятная инструкция запуска.

## Фаза 1. Базовая модель данных

Цель:

- заложить устойчивый фундамент.

Пример промпта:

```text
На основе 01_product_spec.md реализуй базовые модели SQLAlchemy и миграции:
- Course
- Module
- Resource
- ResourceOutlineItem
- Anchor
- Annotation
- ProgressRecord
- StudySession
- Attachment
Сделай аккуратную структуру db-слоя.
После изменений проверь, что миграция создается и база поднимается.
```

Ожидаемый результат:

- модели;
- настройка сессии БД;
- первая миграция;
- схема, соответствующая ТЗ.

## Фаза 2. Курсы и модули

Цель:

- получить первый рабочий пользовательский слой.

Пример промпта:

```text
Реализуй первый рабочий поток:
- dashboard
- список курсов
- создание курса
- страница курса
- добавление модулей
Используй server-rendered шаблоны, без SPA.
Сделай UI простым, но аккуратным.
```

## Фаза 3. Импорт PDF и ресурсы

Цель:

- перейти от пустой оболочки к реальному учебному сценарию.

Пример промпта:

```text
Реализуй импорт PDF в модуль курса:
- загрузка файла
- сохранение в data/uploads
- создание Resource
- извлечение page count
- попытка извлечь PDF outline
- отображение ресурса на странице курса
Если outline не найден, ресурс все равно должен работать.
Проверь сценарий end-to-end.
```

## Фаза 4. Просмотр PDF и навигация

Цель:

- сделать PDF реально пригодным для обучения.

Пример промпта:

```text
Добавь экран просмотра PDF-ресурса:
- встроенный PDF.js
- боковая панель с outline
- переход к странице по клику
- отображение базового статуса ресурса
Сохрани архитектуру простой и совместимой с будущими заметками.
```

## Фаза 5. Заметки и anchor-модель

Цель:

- сделать взаимодействие с материалом осмысленным.

Пример промпта:

```text
Реализуй систему заметок и закладок для PDF:
- bookmark на страницу
- short note на страницу
- хранение anchor
- список заметок ресурса
- общий экран заметок
- переход из заметки обратно к странице
Если точное выделение текста пока сложно, сначала сделай page-level anchor и явно опиши ограничение.
```

## Фаза 6. Прогресс

Цель:

- начать видеть, что пройдено, а что требует повторения.

Пример промпта:

```text
Добавь систему статусов прогресса:
- not_started
- in_progress
- done
- review
- hard
Статусы должны работать для outline item или хотя бы для page-level fallback.
Покажи прогресс на уровне ресурса и курса.
```

## Фаза 7. Таймер и сессии

Цель:

- связать обучение с измеримым временем.

Пример промпта:

```text
Реализуй таймер обучения:
- start
- pause
- resume
- stop
- free mode
- pomodoro mode
- сохранение StudySession
- экран истории
- базовую статистику по дням
```

## Фаза 8. Повторение и фильтрация

Цель:

- сделать систему полезной не только для первого прохождения, но и для возврата к материалу.

Пример промпта:

```text
Добавь экран Review:
- собери туда элементы со статусом review и hard
- покажи фильтры по курсу и ресурсу
- добавь быстрый переход к источнику
```

## Фаза 9. Видео

Цель:

- расширить систему без поломки базовой модели.

Пример промпта:

```text
Добавь поддержку локального видео как Resource:
- загрузка или регистрация локального файла
- HTML5 video player
- bookmark по timestamp
- note по timestamp
- переход из заметки к нужному времени
Сохрани ту же модель Anchor, не делай отдельную несвязанную систему.
```

## Фаза 10. Полировка и стабилизация

Цель:

- убрать шероховатости и подготовить MVP к реальному использованию.

Пример промпта:

```text
Проведи ревизию MVP:
- найди слабые UX-места
- исправь явные архитектурные шероховатости без большого переписывания
- улучши адаптивность
- обнови документацию запуска и структуры проекта
Если видишь технический долг, перечисли его отдельно.
```

## 5. Лучшие шаблоны промптов

## Шаблон A. Реализовать фичу

```text
Прочитай 01_product_spec.md и реализуй <фича>.
Ограничения:
- <ограничение 1>
- <ограничение 2>
Нужно:
- <результат 1>
- <результат 2>
После изменений:
- проверь запуск или тесты
- кратко опиши, что изменил
- перечисли ограничения и компромиссы
```

## Шаблон B. Сначала анализ, потом код

```text
Сначала изучи текущую реализацию <область>.
Пока ничего не меняй.
Покажи короткий план:
- что уже есть
- что отсутствует или сломано
- какой следующий шаг лучше всего сделать
После этого я отдельно подтвержу код.
```

## Шаблон C. Безопасный рефакторинг

```text
Нужно привести в порядок <область> без изменения поведения.
Сфокусируйся на:
- читаемости
- разделении ответственности
- удалении дублирования
Не меняй пользовательский сценарий.
После изменений прогоняй проверки и опиши рискованные места.
```

## Шаблон D. Исправление бага

```text
Исправь баг:
<симптом>

Проверь:
- где воспроизводится
- в чем корневая причина
- как исправить минимально достаточным изменением
После фикса объясни причину бага и как ты проверил исправление.
```

## 6. Как работать через расширение VS Code

Рекомендуемый рабочий процесс:

1. Открывать корень проекта как workspace.
2. Держать `01_product_spec.md`, `02_codex_roadmap.md` и `03_system_overview.md` в видимой части проекта.
3. В начале новой задачи явно писать:
   `Используй 01_product_spec.md как источник истины`.
4. Давать по одной фиче за раз.
5. После каждого важного шага запускать приложение локально, если это уже возможно.
6. Если задача влияет на архитектуру, сначала просить короткий инженерный план.
7. Если изменение затрагивает много файлов, просить краткую сводку и список компромиссов.

## 7. Как избегать слабых результатов

Не стоит использовать слишком расплывчатые запросы:

- "Сделай красиво"
- "Улучши проект"
- "Добавь все необходимое"

Лучше заменять их конкретикой:

- "Переработай dashboard: добавь блок continue learning, очередь review и статистику за сегодня. Маршрутизацию не меняй."
- "Улучши UI формы создания курса, не трогая серверную логику."

## 8. Что полезно спрашивать перед крупной фичей

Перед реализацией полезно дать такой запрос:

```text
Изучи текущую кодовую базу и скажи:
- какие файлы будут затронуты
- какой самый короткий путь до рабочей версии фичи
- какие риски у реализации
Пока код не пиши.
```

Это снижает риск лишних переписываний.

## 9. Когда просить тесты

Тесты лучше просить тогда, когда поведение уже можно формализовать и проверить автоматически.

Хорошие кандидаты:

- агрегация прогресса;
- сервис разбора PDF outline;
- логика создания заметок;
- расчеты таймера и учебных сессий;
- фильтрация очереди повторения.

Пример промпта:

```text
Добавь тесты для сервиса агрегации прогресса.
Покрой обычные сценарии и минимум 2 граничных случая.
Не переписывай production code без необходимости, только если тесты выявят явный структурный дефект.
```

## 10. Как дробить большие задачи

Если задача выглядит слишком крупной, ее нужно дробить явно.

Вместо:

- "Реализуй всю PDF-систему"

Лучше:

1. импорт PDF и метаданных;
2. просмотр PDF;
3. отображение и использование outline;
4. закладки по страницам;
5. заметки по страницам;
6. позже привязки к выделенному тексту.

Так реализация остается проверяемой и управляемой.

## 11. Рекомендуемый цикл работы с Codex

Нормальный рабочий цикл выглядит так:

1. определить следующий точный milestone;
2. попросить Codex изучить код и реализовать только этот milestone;
3. попросить прогнать проверки;
4. посмотреть diff в VS Code;
5. запустить приложение руками;
6. при необходимости попросить доработки;
7. перейти к следующему шагу.

## 13. Как просить обновление документации

Когда архитектура или важная логика меняются, документацию лучше обновлять сразу.

Пример промпта:

```text
Обнови документацию после последних изменений:
- README
- архитектурное описание
- список уже реализованных функций MVP
Меняй только то, что действительно изменилось.
```

## 14. Как просить review

Если нужен именно режим ревью, это лучше обозначать прямо.

Пример:

```text
Сделай review текущей реализации PDF notes.
Ищи в первую очередь:
- баги
- риски регрессии
- пропущенные edge cases
- слабые места в структуре
Ничего не меняй, нужен только вывод по делу.
```

## 15. Что делать, если Codex пытается сделать слишком много

Нужно резко сузить задачу.

Примеры:

```text
Стоп. Сузим задачу.
Сейчас нужен только backend для <фича>.
UI пока не трогай.
```

Или:

```text
Не делай рефакторинг вне зоны задачи.
Меняй только файлы, связанные с <фича>.
```

## 16. Что делать, если сначала нужна архитектура

Можно отдельно запросить проектирование без кода.

Пример:

```text
Пока ничего не реализуй.
На основе 01_product_spec.md подготовь:
- структуру проекта
- модель данных
- ключевые маршруты
- список сервисов
- порядок реализации
Нужен короткий инженерный план перед стартом.
```

## 17. Итоговые рекомендации

- Держать продуктовые решения в `01_product_spec.md`.
- Держать правила работы с Codex в этом файле.
- Предпочитать много маленьких запросов одному огромному.
- Просить рабочие вертикальные срезы, а не абстрактные слои.
- После каждого этапа просить проверку.
- Если фича сформулирована мутно, сначала запускать этап анализа и планирования, а уже потом код.
