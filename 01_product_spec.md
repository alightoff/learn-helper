# Техническое задание: локальная учебная система для книг, видео, заметок и прогресса

## 1. Назначение документа

Этот документ фиксирует полное ТЗ на первую реализацию учебной системы.

Он написан сразу для двух ролей:

- для разработчика, который будет проектировать и реализовывать приложение;
- для AI-ассистента, который будет работать внутри репозитория и вносить изменения по шагам.

Цель документа:

- убрать двусмысленность до начала разработки;
- задать устойчивую архитектурную основу;
- сохранить локальный, расширяемый и универсальный характер системы.

## 2. Видение продукта

Продукт представляет собой локальную учебную среду, которая объединяет:

- учебные материалы;
- структурированные заметки;
- прогресс по темам;
- очередь на повторение;
- таймеры и учебные сессии;
- легкую мотивацию и статистику.

Ключевая идея:

система должна быть не просто хранилищем PDF и видео, а средой обучения, где все действия пользователя привязаны к конкретным учебным единицам.

Примеры учебных единиц:

- для PDF это может быть глава, раздел, страница или выделенный фрагмент;
- для видео это может быть глава, таймкод или временной диапазон.

Любое важное действие пользователя должно иметь привязку к источнику.

## 3. Главная проблема пользователя

Пользователь обучается самостоятельно и сталкивается с типичными трудностями:

- книги, видео и файлы лежат в разных местах;
- заметки оторваны от оригинального материала;
- трудно отслеживать прогресс;
- сложные места нигде системно не отмечаются;
- нет единой очереди на повторение;
- нет истории учебных сессий и статистики по времени;
- неудобно возвращаться из заметки к конкретной странице или таймкоду.

Система должна решить это за счет единого рабочего пространства для самообучения.

## 4. Целевая аудитория

Основной пользователь:

- один человек;
- самостоятельный ученик;
- изучает технические темы: кибербезопасность, реверс-инжиниринг, программирование, математику и не только;
- в основном работает с компьютера;
- позже может заходить в систему с телефона.

Первая версия продукта проектируется как локальная single-user система.

Многопользовательский режим не входит в начальный объем.

## 5. Принципы продукта

### 5.1 Local-first

Система должна полноценно работать без интернета.

Оффлайн должны быть доступны:

- открытие материалов;
- создание заметок;
- ведение прогресса;
- просмотр статистики;
- фильтрация заметок, закладок и очереди повторения.

### 5.2 Универсальная модель контента

Система не должна быть зашита только под книги по кибербезопасности.

Она должна поддерживать:

- любой курс;
- любую тему;
- несколько подтем в рамках одного курса;
- разные типы учебных ресурсов внутри одного модуля.

### 5.3 Anchor-based модель

Заметки, комментарии, закладки и флаги повторения должны крепиться к точному месту в исходном материале.

Примеры привязки:

- страница PDF;
- выделение текста в PDF;
- таймкод видео;
- временной диапазон;
- пункт интерактивного содержания.

### 5.4 Многоуровневый прогресс

Прогресс должен существовать на нескольких уровнях:

- курс;
- модуль;
- ресурс;
- учебная единица.

### 5.5 Расширяемость

Архитектура должна позволять позже добавить:

- OCR для сканированных PDF;
- интервальное повторение;
- облачную синхронизацию;
- полноценную web-версию;
- карточки;
- тесты;
- AI-помощь для разметки и суммаризации.

## 6. Рекомендуемый технический подход

Для первой реализации рекомендуется Python-приложение с локальным web-интерфейсом.

Рекомендуемый стек:

- backend: `FastAPI`;
- слой данных: `SQLAlchemy`;
- база данных: `SQLite`;
- миграции: `Alembic`;
- серверные шаблоны: `Jinja2`;
- интерактивность интерфейса: `HTMX + немного vanilla JavaScript`;
- просмотр PDF в браузере: `PDF.js`;
- парсинг PDF: `PyMuPDF` и/или `pypdf`;
- загрузка файлов: стандартная обработка FastAPI;
- стили: обычный CSS с продуманной системой компонентов;
- статистика: сначала простые серверные расчеты и базовые графики.

Почему именно так:

- Python соответствует желаемому направлению разработки.
- FastAPI дает понятную структуру и позволяет потом без боли превратить приложение в сервис.
- SQLite достаточно для single-user local-first сценария.
- Web-интерфейс внутри локального приложения проще адаптировать позже под телефон, чем классический desktop GUI.
- Jinja2 + HTMX для MVP разумнее, чем тяжелый SPA-фронтенд: меньше сложности, быстрее итерации.

## 7. Объем по версиям

## 7.1 Объем MVP

В MVP обязательно должны быть:

- создание курсов и модулей;
- загрузка PDF;
- добавление локальных видеофайлов;
- дерево курса и ресурсов;
- попытка извлечь содержание PDF;
- ручное создание или исправление структуры, если парсинг не удался;
- открытие PDF и переход к странице из содержания;
- закладки на страницах;
- короткие заметки, привязанные к страницам или выделению;
- статусы единиц: `not_started`, `in_progress`, `done`, `review`, `hard`;
- учет прогресса по единицам с агрегацией вверх;
- таймер учебы и режим Pomodoro;
- история учебных сессий;
- отдельный экран со всеми заметками;
- теги для заметок и закладок;
- вложения к курсу, модулю, ресурсу или единице;
- базовая статистика.

## 7.2 После MVP

Во вторую очередь можно добавить:

- OCR;
- заметки по транскрипту видео;
- привязку к выделенному тексту в PDF на более глубоком уровне;
- очередь интервального повторения;
- серию дней, XP, уровни, достижения;
- адаптацию под мобильный интерфейс;
- экспорт в Markdown или PDF;
- импорт заметок из Obsidian;
- пользовательские шаблоны обучения;
- облачную синхронизацию и публикацию в прод.

## 8. Функциональные требования

## 8.1 Управление курсами

Пользователь должен иметь возможность:

- создать курс;
- задать название, описание, тему, сложность и при желании обложку;
- архивировать и восстанавливать курс;
- видеть краткую сводку по курсу:
  прогресс, количество ресурсов, количество заметок, учебное время.

Каждый курс содержит модули.

## 8.2 Управление модулями

Пользователь должен иметь возможность:

- создавать модули внутри курса;
- менять их порядок;
- в будущем поддержать вложенность, но для MVP это не обязательно.

Для первой версии достаточно плоской структуры модулей внутри курса.

Каждый модуль может содержать:

- ресурсы;
- заметки;
- вложения;
- прогресс.

## 8.3 Управление ресурсами

Поддерживаемые типы ресурсов в MVP:

- `pdf`;
- `video_local`;
- `file_attachment`;
- `link_placeholder` как задел на будущее.

Каждый ресурс должен иметь:

- название;
- тип;
- путь к файлу;
- опциональное описание;
- теги;
- статус;
- даты создания и обновления.

## 8.4 Сценарий работы с PDF

Пользователь должен иметь возможность:

- загрузить PDF в модуль;
- открыть PDF внутри приложения;
- увидеть автоматически извлеченное содержание, если оно есть;
- нажать на пункт содержания и перейти к нужной странице;
- вручную исправить, добавить или отсортировать элементы структуры, если PDF разобран плохо;
- помечать страницы или разделы как пройденные, в процессе, на повторение или сложные;
- выделять текст и создавать заметку, закладку или комментарий;
- открыть общий список заметок и вернуться к точной странице.

Система должна хранить:

- метаданные документа;
- число страниц;
- элементы содержания;
- при необходимости текстовые выдержки для привязки;
- пользовательские аннотации.

## 8.5 Сценарий работы с видео

Пользователь должен иметь возможность:

- добавить локальный видеофайл;
- открыть его в приложении;
- создавать закладки на таймкодах;
- создавать заметки на таймкод или диапазон времени;
- отмечать фрагменты как пройденные, сложные или требующие повторения;
- позднее возвращаться из заметки к нужному моменту.

Если автоматическое извлечение глав недоступно, в MVP допускается ручное создание сегментов.

## 8.6 Заметки и аннотации

В системе должны быть предусмотрены типы аннотаций:

- быстрая заметка;
- закладка;
- комментарий;
- вопрос;
- краткий итог;
- элемент на повторение.

Каждая аннотация должна поддерживать:

- короткий заголовок;
- основной текст;
- теги;
- связь с курсом, модулем и ресурсом;
- опциональную точную привязку;
- даты создания и обновления;
- статус;
- приоритет при необходимости.

Также нужны "глобальные заметки", не привязанные к одной точке, например итог по целой главе.

## 8.7 Отслеживание прогресса

Прогресс должен моделироваться сначала на минимальной учебной единице, а затем агрегироваться выше.

Статусы:

- `not_started`;
- `in_progress`;
- `done`;
- `review`;
- `hard`;
- `skipped` можно добавить позже.

Источники прогресса:

- явная ручная отметка пользователя;
- вспомогательные сигналы, например посещенная страница или просмотренный отрезок, но они не должны считаться окончательной истиной.

Примеры агрегации:

- прогресс ресурса считается по его единицам;
- прогресс модуля по ресурсам и прямым единицам;
- прогресс курса по модулям.

Пользователь всегда должен иметь возможность вручную переопределить вычисленный прогресс.

## 8.8 Учебные сессии и таймеры

Система должна поддерживать:

- ручной старт сессии;
- паузу;
- продолжение;
- завершение;
- типы сессий:
  чтение, видео, повторение, практика, конспект.

Режим Pomodoro должен поддерживать:

- длину рабочего интервала;
- длину короткого перерыва;
- в будущем длинный перерыв;
- число циклов;
- визуальное состояние таймера;
- сохранение истории.

Статистика должна показывать:

- общее время;
- время по дням;
- время по неделям;
- время по месяцам;
- время по курсам;
- время по типам сессий.

## 8.9 Повторение

Должна существовать очередь повторения, куда попадают:

- элементы со статусом `review`;
- элементы со статусом `hard`;
- вопросы;
- позже элементы с датой следующего повторения.

Для MVP достаточно простого списка на повторение.

Планировщик интервального повторения можно добавить позже.

## 8.10 Поиск и фильтрация

Пользователь должен иметь возможность искать по:

- названию курса;
- названию модуля;
- названию ресурса;
- тексту заметки;
- тегам;
- сохраненному фрагменту текста.

Фильтры должны включать:

- статус;
- тип;
- период;
- курс или модуль;
- тип ресурса;
- теги.

## 8.11 Вложения

Пользователь должен иметь возможность прикреплять файлы к:

- курсу;
- модулю;
- ресурсу;
- учебной единице.

Поддерживаемые примеры:

- `.md`;
- `.txt`;
- `.docx`;
- лабораторные файлы;
- фрагменты кода;
- архивы.

В MVP достаточно хранения и отображения списка вложений.

Встроенные редакторы всех форматов пока не нужны.

## 8.12 Статистика и мотивация

Статистика в MVP:

- общее время обучения;
- количество завершенных единиц;
- число заметок;
- число элементов на повторение;
- недавняя активность;
- серия дней при простой реализации.

Геймификация в MVP должна быть легкой:

- процент завершения;
- серия дней;
- число выполненных единиц за сегодня.

Сложную игровую экономику и достижения пока делать не нужно.

## 9. Информационная архитектура

Рекомендуемая основная навигация:

- Dashboard
- Courses
- Course Detail
- Resource Viewer
- Notes
- Review
- Sessions
- Statistics
- Settings

### 9.1 Dashboard

На главной странице должны быть:

- текущие курсы;
- блок "Продолжить обучение";
- размер очереди повторения;
- учебное время за сегодня;
- последние заметки;
- активный таймер или кнопка старта сессии.

### 9.2 Страница курса

На странице курса должны быть:

- метаданные курса;
- список модулей;
- ресурсы внутри модулей;
- прогресс-бары;
- последние заметки по курсу;
- вложения.

### 9.3 Просмотр ресурса

На экране ресурса должны быть:

- название и метаданные;
- дерево содержания или глав;
- текущий статус;
- список заметок и закладок ресурса;
- основная зона просмотра:
  PDF или видео;
- действия для создания заметки, закладки и отметки на повторение.

### 9.4 Экран заметок

Экран заметок должен давать:

- единый индекс всех заметок;
- поиск и фильтры;
- переход назад к источнику.

### 9.5 Экран повторения

Должен показывать:

- все элементы `review` и `hard`;
- переход к источнику;
- возможность снять с повторения или оставить в очереди.

### 9.6 Сессии и статистика

Должны отображаться:

- активный таймер;
- история сессий;
- графики или сводки.

## 11. Стратегия хранения файлов

Рекомендуемая локальная структура хранения:

```text
data/
  app.db
  uploads/
    courses/
      <course_slug>/
        resources/
        attachments/
  previews/
  exports/
```

Требования:

- оригинальные загруженные файлы должны сохраняться;
- исходные файлы нельзя мутировать;
- производные данные хранятся отдельно, в базе или служебных каталогах;
- имена файлов должны санитизироваться;
- удаление и замена файлов позже должны быть безопасными и предсказуемыми.

## 12. Логика импорта и парсинга

### 12.1 Импорт PDF

При импорте PDF система должна:

1. сохранить файл локально;
2. прочитать базовые метаданные;
3. определить число страниц;
4. попытаться извлечь содержание;
5. создать `Resource`;
6. создать `ResourceOutlineItem`, если есть структура;
7. сделать ресурс доступным даже при частичном разборе.

Важно:

- неудачный парсинг не должен ломать импорт;
- пользователь позже может вручную исправить структуру.

### 12.2 Импорт видео

При импорте видео система должна:

1. сохранить путь или скопировать файл согласно настройке;
2. считать метаданные, если это доступно;
3. создать ресурс;
4. при необходимости позволить вручную создать сегменты позже.

## 13. UX-требования

- Интерфейс должен ставить скорость навигации выше декоративности.
- Базовые действия должны быть доступны максимум за 1-2 клика.
- Заметки должны создаваться быстро прямо в процессе чтения или просмотра.
- Возврат из заметки к источнику должен быть мгновенным.
- Прогресс должен быть заметен на всех релевантных экранах.
- Основной приоритет интерфейса это ноутбук и десктоп.
- Адаптация под телефон должна быть заложена в архитектуру, но не мешать desktop-first разработке.

## 14. Нефункциональные требования

- полноценная offline-first работа;
- приемлемая производительность на больших PDF;
- отсутствие жесткой зависимости от интернета;
- сохранность данных между перезапусками;
- возможность резервного копирования через копирование папки данных;
- модульный код, пригодный для дальнейшего перехода в полноценное web-приложение.

## 15. Безопасность и приватность

Это локальное учебное приложение, а не security-продукт.

Для MVP:

- без аккаунтов;
- без удаленной синхронизации;
- все данные остаются локально;
- нужна базовая валидация путей и файлов;
- нельзя допускать произвольное выполнение вложенных файлов;
- именам файлов доверять нельзя.

## 17. Рекомендуемый порядок реализации

## 18. Критерии приемки MVP

MVP можно считать рабочим, если пользователь может пройти следующие сценарии end-to-end.

### Сценарий A. Работа с книгой

1. Создать курс.
2. Создать модуль.
3. Загрузить PDF.
4. Открыть PDF внутри приложения.
5. Увидеть содержание или создать его вручную.
6. Перейти из содержания на нужную страницу.
7. Создать закладку или заметку.
8. Пометить раздел как пройденный или отправить на повторение.
9. Открыть общий список заметок и вернуться к странице-источнику.

### Сценарий B. Работа с видео

1. Добавить видео-ресурс.
2. Открыть видео.
3. Создать закладку по таймкоду.
4. Создать заметку на таймкод.
5. Позже открыть заметку и перейти обратно к нужному моменту.

### Сценарий C. Учет сессий

1. Запустить учебную сессию.
2. Поставить на паузу и продолжить.
3. Завершить сессию.
4. Увидеть ее в истории.
5. Увидеть обновленную статистику по времени.

## 19. Явные нецели MVP

Следующее сознательно не входит в первую версию:

- облачная синхронизация;
- аккаунты пользователей;
- совместная работа;
- полноценный OCR-конвейер;
- AI-суммаризация;
- карточки и тесты;
- сложная игровая экономика;
- plugin-система.

## 20. Заметки для будущего разработчика и ассистента

- Не переусложнять фронтенд, пока не работают базовые учебные сценарии.
- Держать доменную модель устойчивой и расширяемой.
- Рассматривать аннотации PDF и видео как варианты одной anchor-модели, а не как две разрозненные системы.
- Предпочитать явные модели и таблицы, а не хаотичные JSON-поля, кроме действительно гибких данных выделения и метаданных.
- Разрабатывать приложение вокруг реальных пользовательских сценариев, а не вокруг набора изолированных фич.
- При выборе библиотек и архитектуры всегда сохранять local-first характер системы.
