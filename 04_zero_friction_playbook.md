# Playbook без заморочек: как работать с Codex по шагам в VS Code

## 1. Что это за файл

Этот файл нужен для режима "минимум думать про процесс".

Твоя задача:

- открыть проект в VS Code;
- копировать готовые промпты по порядку;
- иногда запускать приложение;
- по сигналу открывать новый чат.

Вся логика работы уже разложена по шагам.

## 2. Базовая идея работы

Самый удобный режим:

- один крупный шаг разработки = один отдельный чат с Codex;
- после завершения шага ты проверяешь результат;
- потом открываешь новый чат и переходишь к следующему шагу.

Так Codex не захлебывается историей, не теряет фокус и не начинает тянуть старый контекст куда не надо.

## 3. Минимальные настройки VS Code

Рекомендуемые настройки рабочего режима:

- открыть именно корень проекта как workspace;
- держать в корне файлы `01_product_spec.md`, `02_codex_roadmap.md`, `04_zero_friction_playbook.md`;
- закрепить `01_product_spec.md` и этот playbook в открытых вкладках;
- использовать встроенный терминал VS Code;
- сохранять файлы автоматически или хотя бы часто;
- если расширение Codex позволяет выбирать режим, использовать обычный coding/agent режим, а не lightweight chat;
- если расширение позволяет подтверждать команды, оставить ручное подтверждение для установки зависимостей и потенциально опасных действий.

Если в расширении есть выбор модели:

- для архитектуры, базы данных, сложных фич и рефакторинга лучше использовать основную сильную coding-модель;
- mini-варианты оставлять для мелких правок и простых вопросов.

## 4. Что всегда держать в голове

Каждый новый чат должен начинаться так, будто Codex ничего не знает, кроме файлов в проекте.

Поэтому в начале почти каждого чата нужно явно писать:

- что за проект;
- какой документ является источником истины;
- какой текущий этап;
- что именно нужно сделать сейчас;
- что не нужно делать.

## 5. Когда открывать новый чат

Открывай новый чат в любом из этих случаев:

- завершен один milestone;
- сделано что-то заметное и уже проверенное;
- чат стал длинным и ответы начали расползаться;
- мы переключаемся с backend на UI;
- мы переключаемся с новой фичи на багфикс;
- Codex начал забывать ограничения или предлагать странные переделки.

Практическое правило:

- почти всегда новый шаг = новый чат.

## 6. Когда НЕ надо менять чат

Можно продолжать текущий чат, если:

- нужно просто добить мелкий дефект после только что сделанного шага;
- нужно поправить 1-2 файла в рамках той же самой фичи;
- проблема обнаружена сразу после последнего ответа и контекст еще свежий.

## 7. Базовый шаблон старта нового чата

Каждый новый чат можешь начинать почти этим текстом:

```text
Работаем над локальным приложением для самообучения на Python.
Источник истины: 01_product_spec.md.
Следуй также 04_zero_friction_playbook.md.

Сначала изучи текущую кодовую базу и только потом реализуй задачу.
Не делай лишний рефакторинг вне зоны текущего шага.
После изменений:
- проверь запуск или тесты, если это возможно
- кратко опиши, что изменил
- перечисли ограничения и компромиссы

Текущий шаг: <вставь шаг ниже>.
```

После этого вставляешь конкретный промпт этапа.

## 8. Правило "один шаг, один результат"

Не проси в одном сообщении:

- и базу;
- и UI;
- и PDF-парсер;
- и видео;
- и статистику.

Это почти всегда ухудшает результат.

Лучше идти так:

1. каркас;
2. база;
3. курсы и модули;
4. PDF;
5. заметки;
6. прогресс;
7. таймер;
8. review;
9. видео;
10. полировка.

## 9. Пошаговый маршрут с готовыми промптами

Ниже идет практически готовая последовательность. Просто бери по порядку.

## Шаг 0. Если проект еще пустой

Если в проекте пока почти ничего нет, начни с этого.

Промпт:

```text
Работаем над локальным приложением для самообучения на Python.
Источник истины: 01_product_spec.md.
Следуй также 04_zero_friction_playbook.md.

Сначала изучи текущую структуру проекта.
Потом создай базовый каркас MVP как локального Python web app.

Стек:
- FastAPI
- SQLite
- SQLAlchemy
- Alembic
- Jinja2
- HTMX

Нужно:
- создать структуру папок
- оформить зависимости
- добавить конфиг
- сделать main.py
- добавить базовый layout шаблон
- сделать README с инструкцией запуска

Не добавляй пока функциональность вне каркаса.

После изменений:
- проверь, что приложение стартует
- кратко опиши структуру проекта
- перечисли, что еще пока не реализовано
```

Когда закончится этот шаг:

- проверь, что проект запускается;
- если все нормально, закрой чат;
- открой новый чат для следующего шага.

## Шаг 1. База данных и модели

Новый чат.

Промпт:

```text
Работаем над локальным приложением для самообучения на Python.
Источник истины: 01_product_spec.md.
Следуй также 04_zero_friction_playbook.md.

Сначала изучи текущую кодовую базу.
Потом реализуй базовые модели базы данных и первую миграцию.

Нужны сущности:
- Course
- Module
- Resource
- ResourceOutlineItem
- Anchor
- Annotation
- ProgressRecord
- StudySession
- Attachment

Сделай структуру db-слоя аккуратной и пригодной для расширения.
Не делай лишний UI.

После изменений:
- проверь, что миграция создается и база поднимается
- кратко опиши итоговую схему
- перечисли спорные места, если они есть
```

После шага:

- если миграции и запуск работают, переходи к новому чату.


## 10. Что делать между шагами

После каждого завершенного шага:

1. прочитай краткий отчет Codex;
2. запусти приложение;
3. проверь руками основной сценарий;
4. если все ок, зафиксируй изменения в git;
5. закрой чат;
6. открой новый чат для следующего шага.

Если руками нашел мелкий дефект, можно:

- либо добить его в этом же чате;
- либо открыть новый чат, если шаг уже реально завершен.

## 11. Когда точно менять контекстное окно

Меняй чат или контекст обязательно, если:

- в истории уже много сообщений;
- Codex начинает забывать требования;
- задача резко сменилась по типу;
- ты закончил один milestone и переходишь к следующему;
- ты хочешь чистую инженерную голову без хвоста старых обсуждений.

Практическое правило:

- после каждого большого завершенного шага лучше новый чат;
- после каждых 2-3 мелких правок внутри одного шага тоже лучше новый чат.

## 12. Что писать в начале нового чата после предыдущего шага

Используй такой короткий "переходный" шаблон:

```text
Продолжаем проект локальной учебной системы.
Источник истины: 01_product_spec.md.
Следуй 04_zero_friction_playbook.md.

Предыдущий этап уже завершен и проверен.
Сначала изучи текущую кодовую базу.
Потом выполни только текущий шаг без лишнего рефакторинга.

Текущий шаг: <вставь название этапа>.
```

## 13. Если хочешь вообще не думать, какой промпт брать дальше

Правило простое:

- открыл новый чат;
- вставил шаблон старта;
- вставил следующий шаг по номеру;
- дождался реализации;
- проверил;
- открыл новый чат.

То есть твоя логика сводится к одному циклу:

`следующий шаг -> копипаст промпта -> проверка -> новый чат`.

## 14. Если нужен только анализ без кода

Иногда лучше сначала спросить так:

```text
Пока ничего не реализуй.
Сначала изучи кодовую базу и скажи:
- какие файлы затронет следующий шаг
- какой самый короткий путь до рабочей версии
- какие риски и компромиссы ты видишь
```

Это полезно перед сложными этапами, например перед заметками, видео или большим рефакторингом.

## 15. Если Codex ушел не туда

Сразу тормози и сужай задачу.

Готовые фразы:

```text
Стоп. Не трогай остальные части проекта.
Исправь только текущую область.
```

```text
Стоп. Сейчас нужен только backend.
UI пока не меняй.
```

```text
Стоп. Пока ничего не кодируй.
Сначала покажи короткий план.
```

## 16. Самый короткий режим работы

Если хочется вообще сверхпростую схему, делай только это:

1. используй шаги 0-10 по порядку;
2. каждый шаг делай в новом чате;
3. после каждого шага запускай приложение;
4. если шаг работает, переходи к следующему;
5. если не работает, в том же чате проси точечный фикс.

Этого уже достаточно, чтобы без лишней организационной нагрузки довести проект до первого рабочего MVP.
